01 — Estrutura

Modelagem do Banco de Dados

O Realtime Database é organizado como uma árvore JSON. Os nós /users e /admin-data são o coração do modelo — o cargo de cada usuário fica em /users/{uid}/role e é consultado pelas Security Rules em tempo real.

{ // raiz do Realtime Database "users": { "uid_alice_123": { "email": "alice@example.com", "displayName": "Alice", "role": "admin", // ← cargo gravado ao criar conta "createdAt": 1714000000000 }, "uid_bob_456": { "email": "bob@example.com", "displayName": "Bob", "role": "user", // ← usuário comum "createdAt": 1714001000000 } }, "admin-data": { // ⚠ somente admin pode ler/escrever "config": { "maintenanceMode": false, "maxUsersAllowed": 1000, "featureFlags": { "betaUI": true } }, "logs": { "log_001": { "action": "user_deleted", "by": "uid_alice_123" } } }, "public-data": { // ✓ todos os autenticados podem ler "announcements": { "ann_001": { "title": "Bem-vindo!", "body": "Sistema online." } } } }

02 — Segurança

Firebase Security Rules

As regras são executadas no servidor Firebase — o cliente nunca tem acesso direto ao banco sem passar por elas. A função auxiliar isAdmin() verifica o nó /users/{uid}/role em tempo real.

database.rules.json 
{
  "rules": {

    // ── Helper: verifica se o usuário logado é admin ──────────────────
    // Nota: funções reutilizáveis não existem no Realtime DB como no
    // Firestore — a expressão precisa ser repetida ou usada inline.

    // ── /users ────────────────────────────────────────────────────────
    "users": {
      "$uid": {

        // Leitura: admin vê tudo; user vê apenas o próprio perfil
        ".read": "auth != null && (
          root.child('users').child(auth.uid).child('role').val() === 'admin'
          || auth.uid === $uid
        )",

        // Escrita: admin escreve qualquer perfil
        // User pode atualizar displayName, mas NÃO pode alterar o próprio 'role'
        ".write": "auth != null && (
          root.child('users').child(auth.uid).child('role').val() === 'admin'
          || (auth.uid === $uid && !newData.child('role').exists())
        )",

        // Validação: campos obrigatórios e tipos corretos
        ".validate": "newData.hasChildren(['email', 'role', 'createdAt'])
          && (newData.child('role').val() === 'admin'
              || newData.child('role').val() === 'user')"
      }
    },

    // ── /admin-data ───────────────────────────────────────────────────
    "admin-data": {
      ".read":  "auth != null &&
        root.child('users').child(auth.uid).child('role').val() === 'admin'",
      ".write": "auth != null &&
        root.child('users').child(auth.uid).child('role').val() === 'admin'"
    },

    // ── /public-data ──────────────────────────────────────────────────
    "public-data": {
      // Qualquer usuário autenticado pode ler
      ".read":  "auth != null",
      // Apenas admin pode escrever
      ".write": "auth != null &&
        root.child('users').child(auth.uid).child('role').val() === 'admin'"
    },

    // ── Tudo o mais: negado por padrão ────────────────────────────────
    "$other": {
      ".read":  false,
      ".write": false
    }

  }
}
admin-data
Acesso exclusivo de Admin
Configurações do sistema, feature flags e logs de auditoria. Se role !== 'admin', o Firebase retorna PERMISSION_DENIED antes de qualquer dado trafegar.
users/{uid}
Isolamento por UID
Um user comum só lê e edita o próprio perfil. A variável $uid nas Rules é comparada com auth.uid em tempo real — sem nenhuma lógica no cliente.
users/{uid}/role
Proteção de Escalonamento
A regra !newData.child('role').exists() impede que um user promova a si mesmo para admin. Apenas um admin já autenticado pode alterar o campo role.
public-data
Leitura Autenticada
Anunciamentos visíveis para todos os usuários logados. Escrita restrita a admin — evita spam e garante integridade do conteúdo público.

03 — Autenticação

Firebase Auth + Cargo

Ao criar uma conta, o cargo é salvo imediatamente em /users/{uid}/role via set() logo após o createUserWithEmailAndPassword().

auth.js — fluxo de registro 
async function registerUser(email, pass, name, role) {
  // 1. Cria o usuário no Firebase Auth
  const cred = await createUserWithEmailAndPassword(
    auth, email, pass
  );

  // 2. Salva o perfil + cargo no Realtime Database
  await set(ref(db, `users/${cred.user.uid}`), {
    email,
    displayName: name,
    role,          // 'admin' | 'user'
    createdAt: Date.now()
  });

  // 3. As Security Rules já protegem a partir daqui
  return cred.user;
}
